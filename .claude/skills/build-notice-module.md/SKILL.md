

# GST Filing Module Workflow

## 1. Module Boundary

The GST Filing Module does **not** handle document upload, scanning, OCR, or raw text extraction.

These activities are handled separately by the  **Document Ingestion Lab** .

The GST Filing Module starts only after the document has already been:

* Uploaded
* Categorized as scanned document or text-based document
* Processed for text extraction
* Parsed into structured data
* Stored in Supabase / PostgreSQL

This module is also **not part of the RAG system** unless explicitly required for fetching knowledge-base formats or rules.

---

## 2. Document Ingestion Lab Output

The Document Ingestion Lab sends the following processed output to the GST Filing Module:

```json
{
  "client_id": "client_identifier",
  "document_id": "document_identifier",
  "document_type": "sales_register | purchase_register | gstr_2b | invoice | challan | other",
  "document_category": "scanned | text_based",
  "extracted_text": "raw extracted text",
  "parsed_data": {},
  "metadata": {
    "file_name": "file.pdf",
    "upload_status": "completed",
    "processing_status": "parsed"
  }
}
```

---

## 3. AI Processing Layer

The extracted and parsed data is sent to the AI Processing Layer.

The AI Processing Layer must:

* Read the parsed document data
* Identify the document type
* Compare the document data against hardcoded filing rules
* Validate whether the document is relevant for GST filing
* Return structured file data for the next module
* Decide whether the document is eligible to move forward in the filing process

The AI must not guess missing information.

---

## 4. Client Data Fetching

After the document is validated, the system fetches client details from Supabase PostgreSQL.

Client data may include:

* Client name
* GSTIN
* Business name
* Filing period
* Registration type
* State code
* Return filing category
* Existing document records
* Previously processed filing data

The fetched client details are combined with the validated document data.

---

## 5. Required Document List Generation

Based on the selected GST filing type, the AI Processing Layer creates the required document list.

Example filing types:

* GSTR-1
* GSTR-3B

The required document list is generated by comparing:

* Selected filing type
* Hardcoded prerequisite rules
* Client details
* Available document metadata in Supabase PostgreSQL

Example hardcoded prerequisite list:

```json
{
  "GSTR_1": [
    "sales_invoice",
    "credit_note",
    "debit_note",
    "customer_master"
  ],
  "GSTR_3B": [
    "sales_register",
    "purchase_register",
    "itc_register",
    "gstr_2b",
    "payment_challan"
  ]
}
```

---

## 6. Document Fetch Request

After generating the required document list, the system sends a fetch request to Supabase PostgreSQL.

The fetch request must retrieve only the documents required for the selected GST filing.

The system must check:

* Whether all required documents are available
* Whether the documents belong to the correct client
* Whether the documents belong to the correct filing period
* Whether parsed data is available
* Whether any required document is missing or incomplete

---

## 7. Missing Document Handling

If any required document is missing or incomplete, the system must stop the filing preparation process.

It must return a structured error.

```json
{
  "status": "blocked",
  "reason": "missing_required_documents",
  "missing_documents": [
    "gstr_2b",
    "payment_challan"
  ],
  "next_action": "Upload the missing documents through the Document Ingestion Lab."
}
```

The GST Filing Module must not continue unless all required documents are available.

---

## 8. AI Data Processing and Issue Detection

Once all required documents are available, the AI Processing Layer processes the complete filing dataset.

The AI must:

* Read all required document data
* Compare values across documents
* Identify mismatches
* Identify missing fields
* Identify filing risks
* Identify unsupported values
* Trace every issue back to the source document
* Provide cited issue references

Each issue must be source-linked.

Example:

```json
{
  "issue_type": "data_mismatch",
  "field": "eligible_itc",
  "problem": "ITC value in purchase register does not match GSTR-2B.",
  "source_documents": [
    "purchase_register",
    "gstr_2b"
  ],
  "recommended_action": "Review ITC reconciliation before filing."
}
```

---

## 9. Issue Resolution Box

Detected issues must be sent to an active issue-resolution box.

The user must be able to:

* View detected issues
* Review source references
* Ask questions through an active chat box
* Approve the issue resolution
* Reject the issue resolution
* Continue only after approval

The system must not proceed to final drafting until the user approves the resolved issues.

---

## 10. AI Drafting Layer

After the user approves the processed filing data and issue resolutions, the data is sent to the AI Drafting Layer.

The AI Drafting Layer must:

* Use only approved processed data
* Use only the selected filing format
* Use only available source documents
* Generate the final structured filing output
* Keep the active chat box available for user clarification
* Send the final draft for approval

---

## 11. Final Approval

The final GST filing output must be sent to the user for approval.

The system must not auto-submit the filing to the GST portal.

The final output should be generated in JSON format.

Example:

```json
{
  "filing_type": "GSTR_3B",
  "client_id": "client_identifier",
  "filing_period": "month_year",
  "status": "draft_ready_for_approval",
  "fields": {
    "outward_taxable_supplies": 0,
    "eligible_itc": 0,
    "tax_payable": 0
  },
  "issues_resolved": [],
  "source_documents_used": [
    "sales_register",
    "purchase_register",
    "gstr_2b",
    "payment_challan"
  ],
  "approval_required": true
}
```

---

## 12. Strict Non-Goals

This module must not perform:

* File upload
* OCR
* Raw text extraction
* Document ingestion
* GST portal submission
* Auto-filing
* Payment processing
* Client billing
* WhatsApp integration
* Auto email reminders
* MCA filing
* Income tax filing
* Advanced analytics
* Firm management system
* Mobile app workflow

These are outside the current GST Filing Module scope.

---

## 13. Core Rule

The GST Filing Module must only process already-ingested document data, validate required filing documents, detect issues, support user review, generate structured JSON filing output, and send the final draft for approval.

It must not overbuild beyond this scope.
